# Milestone 8 â€” Game Feel: Waves, Pooling, Score, Restart, Audio

Turns the wired core loop into a complete round: enemies arrive in
**escalating waves** from the arena edges, **killing an enemy scores
points**, player death **shows the Game Over panel** with the final score
still on screen, and **pressing R restarts**. Bullets and enemies are
**pooled** instead of Instantiate/Destroy, and **audio** adds fire / hit /
death SFX plus music. The demo win: on **Play**, survive escalating waves,
watch the score climb, die, see Game Over, press R, and go again.

This is a **pure-code change**. The scene, prefabs, HUD, and game-over panel
already exist and are fully wired â€” do **not** rebuild, re-wire, or edit any
`.unity` / `.prefab` / `.asset` / `.inputactions` file. No new HUD text
elements this milestone (health/wave readouts are a later supervised HUD
pass); the existing score text and game-over panel are reused as-is.

## Context

The arena shooter core loop is already playable and fully wired on develop
(`Assets/Scenes/Main.unity`, build index 0). **Build on the as-built C#
surface below â€” do not create parallel systems:**

- `Gameplay/GameManager` already owns run state: `int Score`,
  `void AddScore(int)` (already a no-op once game over), `bool IsGameOver`,
  `void TriggerGameOver()`. **Extend this** â€” do **not** introduce a separate
  `RunState`/score owner, and do **not** rename or remove these members
  (`PlayerController`, the HUD, and existing tests depend on them).
- `UI/HUDScoreDisplay` already renders `GameManager.Score` every frame;
  `UI/HUDManager` wires it; `UI/GameOverPanel` toggles a hidden panel
  (`Show`/`Hide`) and reloads the scene on its restart button. **Extend
  these** â€” do **not** author a new `HudView`.
- `Gameplay/Enemy` (`Game.IDamageable`; serialized `maxHealth`/`moveSpeed`;
  `Configure(maxHealth, moveSpeed)`; private `Die()`; resolves the player via
  `FindFirstObjectByType<PlayerController>()` in `Awake`), `Gameplay/PlayerController`
  (serialized `maxHealth`, private `currentHealth`, `TakeDamage(float)`,
  `Die()` â†’ `GameManager.TriggerGameOver()`), `Gameplay/EnemySpawner`
  (continuous spawn up to `maxEnemies`, random-edge, `StartSpawning`/
  `StopSpawning`/`IsSpawning`/`Configure`), `Gameplay/Weapon` + `Bullet`
  (Instantiate/Destroy per shot, trigger combat) all already exist.
- A prefab cannot hold a reference to a scene object, so **resolve scene
  singletons at runtime** via `FindFirstObjectByType<T>()` â€” the established
  pattern â€” never an Inspector field that would need scene wiring.

Hard constraints (`.swizz/context.md`): pure logic in the `Game` assembly
(EditMode-tested), MonoBehaviours only wire it; **trigger-based combat**;
**kinematic** movement; **new Input System only**; **no scene/prefab/asset
edits**; all changes inside the existing `Game`, `Gameplay`, and `UI`
assemblies (no new assemblies).

## Features

### Feature: Wave plan (Game, pure logic)

- [ ] `Assets/Scripts/Game/WavePlan.cs`: for a 1-based wave index returns
      enemy count, spawn interval, and enemy speed â€” count and speed
      non-decreasing and interval non-increasing as the wave rises.
- [ ] `Assets/Scripts/Game/EdgeSpawnMath.cs`: maps a normalized value in
      `[0, 1)` to a point on the `Game.ArenaBounds` perimeter.
- [ ] EditMode tests (`Assets/Tests/EditMode/`): count and speed are
      non-decreasing across waves and interval non-increasing; wave 1 returns
      sane starting values; every `EdgeSpawnMath` result lies on the arena
      boundary.

### Feature: Wave-driven spawning (Gameplay â€” modify existing EnemySpawner)

- [ ] `Assets/Scripts/Gameplay/EnemySpawner.cs` drives `WavePlan`: spawns the
      current wave's enemy count at perimeter points (`EdgeSpawnMath`) at the
      wave's interval, applies the wave's speed to each enemy via the existing
      `Enemy.Configure`, and advances to the next wave once every enemy in the
      wave is dead.
- [ ] Exposes the current wave (e.g. `int CurrentWave`) and stops spawning
      once the game is over. Preserve the existing public members
      (`StartSpawning`/`StopSpawning`/`IsSpawning`/`Configure`).
- [ ] A PlayMode test asserts the spawner emits the wave-1 count and that
      clearing a wave advances the wave index.

### Feature: Object pooling (Gameplay)

- [ ] `Assets/Scripts/Gameplay/Pool.cs`: a reusable pool that recycles bullet
      and enemy instances (deactivate + return, reactivate + reuse) instead of
      `Instantiate`/`Destroy` on every shot and spawn.
- [ ] `Weapon`/`Bullet` and `EnemySpawner`/`Enemy` use the pool: a despawned
      bullet (on hit or lifetime) and a dead enemy return to the pool rather
      than `Destroy`. Existing behaviour (damage, trigger combat, chase) is
      unchanged and existing PlayMode tests stay green (update any that assert
      on `Destroy`).
- [ ] A test asserts that returning an instance and getting one again reuses
      the same object instead of allocating a new one.

### Feature: Score on enemy death (Gameplay)

- [ ] `Enemy` gains a serialized `scoreValue` (`int`, default `10`), settable
      by tests (extend the existing `Configure`).
- [ ] On death by damage (inside the existing `Die()`), `Enemy` adds
      `scoreValue` to the scene's `GameManager` exactly once via `AddScore`,
      resolving it at runtime with `FindFirstObjectByType<GameManager>()`. No
      points when an enemy is removed by spawner cleanup, and none after game
      over (rely on `AddScore`'s existing guard â€” don't duplicate it).
- [ ] A PlayMode test: damaging an enemy to death raises `GameManager.Score`
      by `scoreValue`; no score is added after `TriggerGameOver()`.

### Feature: Game-over presentation & restart (Gameplay + UI â€” no new scene objects)

- [ ] `UI/HUDManager` watches `GameManager.IsGameOver` and shows the
      already-wired `GameOverPanel` exactly once when the run ends. The live
      score text stays on screen, so the final score remains visible â€” add no
      new text element.
- [ ] Restart without editing the input asset: a component reads the new Input
      System keyboard (`UnityEngine.InputSystem.Keyboard.current.rKey`) and,
      **only when the game is over**, reloads the active scene
      (`SceneManager.LoadScene(SceneManager.GetActiveScene().buildIndex)`). The
      existing restart-button path on `GameOverPanel` stays as-is.
- [ ] A PlayMode test: forcing game-over activates the game-over panel; the
      restart hook invokes the scene-reload path (inject/spy the reload action
      â€” do **not** assert on a real scene load).

### Feature: Audio (Gameplay â€” runtime AudioSources, no scene wiring)

- [ ] `Assets/Scripts/Gameplay/Sfx.cs` centralizes one-shot playback for fire,
      enemy-hit, enemy-death, and player-hit, plus a looping music source.
      `AudioSource`s are created in code (`AddComponent` at runtime) so no
      scene/prefab edits are needed; clips may be left unset (the hook firing
      is what matters).
- [ ] `Weapon` (fire), `Enemy` (hit + death), and `PlayerController` (hit)
      route their SFX through `Sfx`.
- [ ] A PlayMode test asserts firing the weapon invokes the fire-SFX hook
      (verified by the call, not by sound).

## Architecture Notes

Wave escalation, perimeter math, and any score/wave bookkeeping that can be
pure live in the `Game` assembly and are EditMode-tested; the MonoBehaviour
layer only drives them. Keep `GameManager`'s existing public surface
(`Score`/`AddScore`/`IsGameOver`/`TriggerGameOver`) intact and additive.
Restart is `SceneManager.LoadScene` of the single scene. Audio verification
stays mechanical (an `AudioSource` exists and the right hook fires) â€” never
assert on sound output.

## Out of Scope

Any `.unity` / `.prefab` / `.asset` / `.inputactions` edit; new HUD text for
health or wave number (a later supervised HUD pass); real sprite art, shader /
particle VFX, the main-menu scene, and multiple enemy types (M3).

## Dependencies

The wired arena core loop on develop (player, weapon, bullet, enemy, spawner,
`GameManager`, HUD, `GameOverPanel`).
