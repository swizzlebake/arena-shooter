# Milestone 2 — Game Feel: Waves, Health, Score, Game Over

Turns the M1 nucleus into a complete round. Enemies arrive in escalating
waves from the arena edges, the player has health shown in a HUD, killing
enemies scores points, and player death shows a Game Over panel with the
final score and a Restart prompt. Object pooling replaces per-shot
Instantiate/Destroy, and audio adds fire/hit/death SFX plus music. Art is
still placeholder squares — the look-and-feel pass is M3.

## Context

Builds directly on M1 (player, weapon, bullet, enemy, trigger combat) in
the same scene. The `EnemySpawner` and `GameManager` anchors already
exist in `Main.unity` for the spawn director and run manager. Keep the
hard constraints from `.swizz/context.md`: pure logic in the `Game`
assembly (EditMode-tested), MonoBehaviours only wire it, trigger-based
combat, kinematic movement, new Input System only.

## Features

### Feature: Wave plan (Game, pure logic)

- [ ] `Assets/Scripts/Game/WavePlan.cs`: for a 1-based wave index returns
      enemy count, spawn interval, and enemy speed, each escalating
      monotonically with the wave.
- [ ] `Assets/Scripts/Game/EdgeSpawnMath.cs`: maps a normalized value to
      a point on the arena perimeter.
- [ ] EditMode tests: count and speed are non-decreasing across waves;
      wave 1 returns sane starting values; every `EdgeSpawnMath` result
      lies on the arena boundary.

### Feature: Spawn director (Gameplay)

- [ ] `Assets/Scripts/Gameplay/EnemySpawner.cs` on the existing
      `EnemySpawner` object drives `WavePlan`, spawning enemies at
      perimeter points at the wave's interval, and advances to the next
      wave once every enemy in the current wave is dead.
- [ ] A PlayMode test asserts the spawner emits the wave-1 count and that
      clearing the wave advances the wave index.

### Feature: Object pooling (Gameplay)

- [ ] A reusable pool (`Assets/Scripts/Gameplay/Pool.cs`) recycles bullet
      and enemy instances instead of Instantiate/Destroy on every shot
      and spawn.
- [ ] A test asserts that returning an instance and getting again reuses
      the same object rather than allocating a new one.

### Feature: Run state, score & game-over (Game + Gameplay)

- [ ] `Assets/Scripts/Game/RunState.cs` (pure): `Score`, `Wave`,
      `IsGameOver`; `AddKill(points)`, `EndRun()`. EditMode-tested.
- [ ] `Assets/Scripts/Gameplay/GameManager.cs` on the existing
      `GameManager` object owns a `RunState`, adds score on enemy death,
      ends the run on player death (stopping spawns), and restarts on the
      Input System "restart" (R) action by reloading `Main.unity`.
- [ ] PlayMode tests: killing an enemy increases the score; player health
      reaching 0 sets game-over and halts spawning; restart reloads to a
      fresh run (score 0, wave 1).

### Feature: Audio (Gameplay)

- [ ] `AudioSource`-based one-shots for fire, enemy hit, enemy death, and
      player hit, plus a looping music track on the `GameManager` (or a
      dedicated audio object).
- [ ] `Assets/Scripts/Gameplay/Sfx.cs` centralizes one-shot playback so
      all SFX route through one place. Clips may be placeholder tones.
- [ ] A PlayMode test asserts firing the weapon invokes the fire-SFX
      hook (verified by the call, not by sound).

### Feature: Scene integration

- [ ] On Play, `Main.unity` runs a full round: `EnemySpawner` drives
      waves, `GameManager` owns run state + restart, the HUD + game-over
      panel are present and wired, and audio sources exist.
- [ ] A PlayMode smoke test plays until game-over is reachable (player
      takes lethal damage), asserts the game-over panel shows the final
      score, then asserts restart returns to a live wave-1 run.

## UI Features

### UI Feature: Heads-up display (UI assembly, uGUI)

- [ ] `Assets/Scripts/UI/UI.asmdef` references `Game`.
- [ ] `Main.unity` gains a uGUI `Canvas` with `Assets/Scripts/UI/HudView.cs`
      showing current/max health, score, and wave number, updating from
      `RunState` and the player's health.
- [ ] A Game Over panel (hidden during play) activates on game-over,
      showing the final score and a "Press R to restart" prompt.
- [ ] A PlayMode test drives a kill and asserts the score text updates;
      forces game-over and asserts the panel becomes active with the
      final score.

## Architecture Notes

Spawning, wave escalation, and scoring are pure `Game` logic exercised by
EditMode; the MonoBehaviour layer only wires them. Restart is
`SceneManager.LoadScene` of the single scene. The HUD uses uGUI
(`com.unity.modules.ui`, already in the manifest). Audio verification
stays mechanical (an `AudioSource` exists and the right hook fires) — do
not assert on sound output.

## Out of Scope

Real sprite art, shader/particle VFX, the main-menu scene, and multiple
enemy types — all M3.

## Dependencies

M1 (player, weapon, bullet, enemy, trigger combat) in the same scene.
