# Milestone 05 â€” Breakout: Paddle, Ball & Walls

The playable core of a 2D side-view Breakout clone, built with Unity
best practices and thoroughly covered by both EditMode and PlayMode
tests. A single player slides a paddle horizontally â€” bounded by the
screen edges â€” while a constant-speed ball bounces off the paddle and
the left, right, and top walls. Blocks, breaking, and the win/lose
round arrive in Milestone 06; this milestone delivers the loop those
features hang on, and the decoupled logic seam that makes the whole
thing unit-testable.

The design is **logic-seam-first**: all collision math (paddle
clamping, ball reflection) lives in a `MonoBehaviour`-free logic
assembly, so it runs in EditMode without entering Play. That assembly
may freely use UnityEngine value types (`Vector2`, `Mathf`) â€” it simply
never derives from `MonoBehaviour` or touches scene/Play APIs. The
MonoBehaviour layer is a thin shell over that seam and is exercised by
PlayMode integration tests.

Run with:

```bash
swizz run --profile unity examples/milestones/05-unity-breakout.md
```

A Unity Editor instance must be open on the target project with the
[IvanMurzak/Unity-MCP](https://github.com/IvanMurzak/Unity-MCP) package
installed; `swizz doctor --profile unity` surfaces the host-side
prerequisites.

## Context

- Unity 6000.x, 2D (Built-in Render Pipeline), C# only. The Unity
  project is the repo root (`Assets/...`); adjust the paths if your
  project lives in a subdirectory.
- Game logic must be `MonoBehaviour`-free for testability â€” it lives in
  its own assembly. Using UnityEngine APIs (`Vector2`, `Mathf`, etc.)
  is fine; the only rule is no `MonoBehaviour` subclasses and no
  scene/Play-only calls, so it loads in EditMode without entering Play.
- The new Input System package (`com.unity.inputsystem`) only; the
  legacy `UnityEngine.Input` class is forbidden. Set Project Settings â†’
  Player â†’ Active Input Handling to **Input System Package (New)** (or
  **Both** if UGUI default events need it).
- Physics is 2D (`com.unity.modules.physics2d`). Collisions are
  detected with 2D colliders; the **direction** of every bounce is
  decided by the pure logic seam, not left to physics restitution, so
  the ball keeps a constant arcade speed.
- **Primitive visuals only** â€” solid-color sprites via the built-in
  unit sprite (or a runtime-generated `Texture2D`) tinted per object;
  the ball uses a built-in circle/knob sprite. No imported `.png` art,
  so the Artist agent is skipped.

## Features

### Feature: Playfield geometry & collision math

A `MonoBehaviour`-free assembly that owns the playfield bounds and the
math the gameplay layer leans on: clamping the paddle inside the field
and reflecting the ball off a surface. Self-contained so it unit-tests
in EditMode without Play, physics, or a scene. The well-trodden parts
(axis-aligned reflection) should be treated as a port, not a fresh
design.

**Acceptance Criteria:**
- [ ] `Assets/Scripts/Game/Playfield.cs` defines a `Playfield` with
      `Left`, `Right`, `Top`, `Bottom` bounds and a `Contains(point)`
      / in-bounds check
- [ ] `Assets/Scripts/Game/Paddle.cs` (or equivalent) exposes
      `ClampX(float x, float halfWidth)` that keeps the paddle centre
      within `[Left + halfWidth, Right - halfWidth]`
- [ ] A reflection helper reflects a velocity across a vertical wall
      (flip X, preserve Y) and a horizontal wall (flip Y, preserve X),
      and a paddle helper steers the bounce by contact offset: a hit at
      the paddle centre returns straight up, a hit near an edge angles
      outward â€” speed **magnitude is conserved** in every case
- [ ] The `Game` assembly is `MonoBehaviour`-free: it may use
      `UnityEngine` types like `Vector2` and `Mathf`, but defines no
      `MonoBehaviour` subclasses and makes no scene/Play-only calls, so
      it loads in the EditMode test assembly without entering Play
- [ ] EditMode tests under `Assets/Tests/EditMode/` cover: clamp stops
      exactly at the left and right edges; right-wall reflect flips the
      X component and preserves Y + speed; top reflect flips Y;
      paddle-centre hit goes straight up; paddle-edge hit angles
      outward; speed magnitude unchanged across each reflection

**Notes:**
- Keep the public surface minimal â€” the gameplay layer should not need
  `Game` internals, only these helpers.
- Express speed conservation as an assertion on vector magnitude with a
  small epsilon, not on exact component equality.

### Feature: Paddle control (keyboard + mouse)

The player paddle, driven by the new Input System through **both**
keyboard (A/D and the left/right arrows) and the mouse (paddle centre
follows the pointer's X). Movement is bounded by the screen edges via
the `Playfield.ClampX` seam, so the paddle physically cannot leave the
field by either input path.

**Acceptance Criteria:**
- [ ] `Assets/Scripts/Gameplay/PaddleController.cs` is a
      `MonoBehaviour` that reads an Input System **Move** action (bound
      to A/D + Left/Right arrows) and the pointer position
- [ ] Keyboard input moves the paddle horizontally at a serialized
      `MoveSpeed` (frame-rate independent â€” multiply by
      `Time.deltaTime`); mouse movement sets the paddle's X to the
      pointer's world X
- [ ] Every resulting X is passed through `Playfield.ClampX` so the
      paddle never crosses the left or right boundary, regardless of
      input source
- [ ] Input is routed via the Input System (`InputAction` asset or
      generated C# class) â€” no `UnityEngine.Input` calls anywhere
- [ ] A PlayMode test under `Assets/Tests/PlayMode/` drives the paddle
      with synthesized keyboard input (via `InputTestFixture`) and
      asserts it moves the correct direction; a second test sets the
      pointer beyond the field edge and asserts the paddle stops
      exactly at the boundary

**Notes:**
- Use `InputTestFixture` / `InputSystem` test APIs to feed virtual
  devices in PlayMode rather than poking real hardware state.
- Test the clamp boundary through the public behaviour, not by reaching
  into private fields â€” the seam is already unit-tested in EditMode.

### Feature: Ball movement & bouncing

A destructive ball (it will break blocks in m06) that travels at a
constant speed and bounces off the paddle and the left, right, and top
walls. The ball's collider detects contacts; the **new direction** of
each bounce comes from the `Game` reflection seam so the speed stays
constant and the behaviour is deterministic. The ball is allowed to
fall past the bottom â€” the loss condition is handled in Milestone 06.

**Acceptance Criteria:**
- [ ] `Assets/Scripts/Gameplay/Ball.cs` is a `MonoBehaviour` with a
      `CircleCollider2D`; the ball moves at a serialized constant speed
      and launches upward at a slight angle when Play starts
- [ ] On contact with a wall or the paddle, the ball's velocity is
      replaced by the reflection-seam result for that surface â€” the
      paddle bounce uses the contact offset to steer the angle
- [ ] Ball **speed magnitude is unchanged** after any bounce (no energy
      gained or lost from physics restitution); whether the ball is
      kinematic with manual reflection or a `Rigidbody2D` with zero
      gravity and zero drag is the architect's call, but constant speed
      is a hard requirement
- [ ] The ball passes freely below the bottom edge (no bottom wall in
      this milestone)
- [ ] PlayMode tests assert: a ball aimed at the right wall reflects
      (X component flips); a ball landing on the paddle reverses its
      vertical direction; and ball speed magnitude is identical before
      and after a bounce

**Notes:**
- Resolve the bounce in `FixedUpdate` / the physics step so contacts
  aren't missed at high speed; consider continuous collision detection
  on the ball collider.
- Don't rely on `Physics2D` bounciness material for direction â€” route
  it through the seam so the PlayMode test and EditMode test agree.

### Feature: Breakout scene & launch wiring

A single scene that assembles the core loop: an orthographic camera
framing the field, static wall colliders on the left, right, and top,
the paddle, and the ball that launches on Play. This is the
scene/prefab wiring the `unity-editor-engineer` performs over MCP, on
top of the compiled C# the engineer authored.

**Acceptance Criteria:**
- [ ] `Assets/Scenes/Breakout.unity` contains an orthographic camera
      sized to frame the playfield, plus left/right/top static wall
      `GameObject`s with 2D colliders matching the `Playfield` bounds
- [ ] A `Paddle` GameObject (primitive solid-color sprite) sits near
      the bottom with `PaddleController` attached; a `Ball` GameObject
      (built-in circle sprite) sits above it with `Ball` attached
- [ ] On entering Play, the ball launches upward and bounces
      indefinitely off the paddle and the three walls
- [ ] Build Settings include only the `Breakout` scene
- [ ] A PlayMode smoke test enters Play, waits ~2 seconds, and asserts
      the ball is still inside the playfield bounds and the console
      logged no errors

**Notes:**
- Render primitives with the built-in unit white sprite (or a
  runtime-generated solid-color `Texture2D`) tinted per object â€” no
  imported textures.
- The wall colliders' positions must agree with the `Playfield` bounds
  the gameplay layer uses, or the ball will visibly clip or bounce
  early.

### Feature: Assembly definitions & test structure

The assembly layout that enforces the decoupling at compile time and
gives the test runner clean EditMode and PlayMode targets.

**Acceptance Criteria:**
- [ ] `Assets/Scripts/Game/Game.asmdef` keeps its `references` array
      empty and does **not** set `noEngineReferences` â€” Unity links
      `UnityEngine`/`UnityEditor` automatically, so the assembly can use
      `Vector2`/`Mathf` while holding only `MonoBehaviour`-free logic.
      Do **not** put `"UnityEngine"` in `references`; that array is for
      other asmdefs only
- [ ] `Assets/Scripts/Gameplay/Gameplay.asmdef` references `Game` and
      `Unity.InputSystem` and holds the MonoBehaviours
- [ ] `Assets/Tests/EditMode/EditModeTests.asmdef` references **only**
      `Game`, with `optionalUnityReferences: ["TestAssemblies"]` and
      `includePlatforms: ["Editor"]`
- [ ] `Assets/Tests/PlayMode/PlayModeTests.asmdef` references `Game` and
      `Gameplay` plus the test assemblies, with no platform restriction
- [ ] Both test assemblies compile and their suites are discoverable by
      the Unity Test Runner (EditMode and PlayMode tabs)

## Architecture Notes

- `game-architect` should split the work explicitly: the **default
  engineer** authors the `MonoBehaviour`-free `Game` assembly and the
  MonoBehaviour `.cs` / `.asmdef` files through the staging layer; the
  **unity-editor-engineer** wires the scene (camera, walls, paddle,
  ball, component references) via MCP `scene.write` / `prefab.write`
  rather than hand-editing `.unity` YAML.
- The engineer's C# must pass the pre-merge compile gate
  (`assets-refresh`) before the scene wiring runs â€” a script with a
  compile error can't be attached to a GameObject.
- Keep the ball's bounce resolution in the physics step and route
  direction through the `Game` seam so the EditMode reflection tests and
  the PlayMode bounce tests are testing the same contract.
- New `.cs` / `.asmdef` files need an asset refresh before Unity sees
  the new types/assemblies; the wiring step depends on that refresh
  having imported them (and generated their `.meta` GUIDs).
