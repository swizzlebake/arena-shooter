# Milestone 02 — Scene wiring: camera + input

Milestone 01 shipped the Game-of-Life model, view, art, and HUD, but its
`Main.unity` is not actually runnable: it has **no Main Camera** (so the
world-space cell/background sprites never render — you see only the
overlay UI), the **click-to-paint input is never delivered** (the
`GridInputHandler` callbacks are wired to nothing), and the failed
scene-config retries left **duplicate `GridView` and `GridInputHandler`
components** in the scene. This milestone finishes the scene so the game
runs end-to-end in the Editor and a standalone PC build.

This is additive forward work on top of the released m01 (`v1.0.0`) — it
must NOT recreate the model/view/HUD that already exist; it only adds the
camera, wires input for real, and de-duplicates the scene.

## Context

- Unity 6000.3.10f1, 2D (Built-in Render Pipeline), C# only. PC /
  Editor-playmode target (no mobile).
- Project lives at `unity-project/`. The Editor is auto-launched by Swizz;
  MCP attaches via the unity-mcp server. Scene/prefab/serialized-ref
  changes go through the editor-engineer's `wiring_plan` (the host
  synthesizes + verifies the C#); plain `.cs` is the C# Engineer's lane.
- The single scene is `Assets/Scenes/Main.unity`; the input asset is
  `Assets/InputActions/Controls.inputactions` (action map `Player`,
  actions `Click` (Button, `<Mouse>/leftButton`) and `Position` (Value,
  `<Mouse>/position`)).
- A runtime stopgap `Assets/Scripts/View/CameraRig.cs` currently creates a
  camera at load if none exists. It is a band-aid to be removed once the
  scene owns a real camera.

## Features

### Feature: Main Camera in the scene

Add a single fixed orthographic Main Camera to `Main.unity` so the
world-space cell grid and background actually render, and so
`Camera.main` exists for screen→world input conversion.

**Acceptance Criteria:**
- [ ] `Main.unity` contains exactly one `Camera` component on a GameObject
      tagged `MainCamera` (so `Camera.main` resolves)
- [ ] The camera is orthographic, positioned to centre the 32×24 grid,
      with an `orthographicSize` that frames the whole board, and a solid
      background colour `#0A0A1F` (the palette the cell art is tuned for)
- [ ] Entering Play in the Editor shows the cell grid + background tile
      filling the view (not a blank/overlay-only screen)
- [ ] The runtime stopgap `Assets/Scripts/View/CameraRig.cs` (and its
      `.meta`) is removed once the scene camera exists — the camera is
      authored in the scene, not synthesized at load
- [ ] No second camera is introduced (exactly one `MainCamera` in the
      scene)

### Feature: Working click-to-paint input

Make the existing `GridInputHandler` actually receive pointer input so the
player can paint live cells, restoring the m01 "Click-to-paint" feature's
behaviour that never got wired in the scene.

**Acceptance Criteria:**
- [ ] In Play mode, left-clicking a cell in the grid toggles it (dead→alive,
      alive→dead); the painted cell is visible immediately
- [ ] Click-and-drag paints continuously, each cell flipped at most once
      per stroke (drag back over a flipped cell does not bounce it)
- [ ] Painting uses the new Input System via `Controls.inputactions` — no
      `UnityEngine.Input.GetMouseButton`; world→grid conversion uses
      `Camera.main.ScreenToWorldPoint`; clicks outside the grid are no-ops
- [ ] The delivery path is robust without a hand-authored UnityEvent:
      `GridInputHandler` self-subscribes to the `Click`/`Position` actions
      (e.g. via the generated C# `Controls` class) in `OnEnable` and
      unsubscribes in `OnDisable`, OR a `PlayerInput` is wired such that
      the callbacks demonstrably fire — proven by the PlayMode test below
- [ ] A PlayMode test under `Assets/Tests/PlayMode/` simulates a pointer
      press over a known grid cell and asserts that cell's model state
      toggled (the handler's callback ran and reached `Grid.Set`)

### Feature: De-duplicate scene wiring

The failed m01 scene-config retries left two `GridView` and two
`GridInputHandler` components in `Main.unity`. Collapse to one of each so
there is a single authoritative grid.

**Acceptance Criteria:**
- [ ] `Main.unity` contains exactly one `View.GridView` (on `GridViewHost`)
      and exactly one `Input.GridInputHandler` (on `InputManager`)
- [ ] The surviving `GridInputHandler.gridView` references the surviving
      `GridView`, and `GameUIController.gridView` references the same one
- [ ] No orphaned/duplicate GameObjects remain from the removed components
- [ ] The background quad's `SpriteRenderer.sprite` references
      `Assets/Sprites/grid-bg-tile.png` and covers the camera view

## Architecture Notes

- Camera + scene-object changes are the editor-engineer's lane via the
  `wiring_plan`; the GameObject **tag** (`MainCamera`) is not a serialized
  field, so tagging is the allowed `script-execute` camera tweak.
- Prefer **self-subscription** for input: enable "Generate C# Class" on
  `Controls.inputactions` and have `GridInputHandler` own a `Controls`
  instance, subscribing `Player.Click` / `Player.Position` in `OnEnable`.
  This keeps input delivery in the C# Engineer's lane and avoids depending
  on a scene-authored `PlayerInput` UnityEvent (which the declarative
  `wiring_plan` cannot author).
- De-duplication: identify the redundant `GridView` / `GridInputHandler`
  components (and any host GameObjects the retries spawned) and remove the
  extras, keeping the pair referenced by `GameUIController`.
- Do NOT modify `Grid.cs`, the cell prefab, the sprites, or the HUD
  buttons — they are released and correct.

## Out of Scope

- Touch / mobile input — PC + Editor playmode only (mouse bindings stay).
- Pattern presets, runtime grid resize, save/load, audio, camera pan/zoom.
- Any change to the m01 model/view/art/HUD beyond the wiring above.

## Dependencies

- Unity 6000.3.10f1 installed (verify with `swizz doctor`).
- Built on the released m01 (`v1.0.0`); `develop` must be at or ahead of
  `main` with no unrelated in-flight work before running.
- The unity-mcp server package present in `Packages/manifest.json` (MCP
  tool calls time out without it).
- `ANTHROPIC_API_KEY` set if `[cloud].always = true`.
